# PROJECT CONTEXT — MINDMESH / Orbital 2026

## Project Overview

MINDMESH is an AI-assisted learning system designed to transform unstructured learning from assignments, tutorials, and practice papers into structured, reusable, and interconnected knowledge.

Core problem being solved:

Students often learn the most from:
- assignments
- tutorials
- practice papers
- mistakes made while solving questions

However, this learning is usually:
- written messily on paper
- scattered across apps/platforms
- not structured for later revision
- not connected across topics

This causes:
- repeated mistakes
- forgotten insights
- inefficient revision
- high manual note-taking overhead

MINDMESH solves this by automating the pipeline:

Raw practice learning
→ rough notes
→ AI structured flashcards
→ topic aggregation
→ conceptual linking
→ revision + sharing

---

# Core Idea

Users upload:
- screenshots/questions from coursework/practice
- rough notes describing what they learnt

Example:
"I forgot to account for edge case where BST is empty"

AI processes this into:
- readable flashcards
- extracted metadata
- topic tags
- related topics

This enables:
1. Coursework flashcards (CFCs)
2. Topical flashcards (TFCs)
3. Mindmap visualisation
4. Sharing system

---

# Terminology

## CFC = Coursework Flashcard

Definition:
A flashcard generated directly from a specific piece of coursework or practice.

Examples:
- tutorial question notes
- assignment mistake
- practice paper insight

Characteristics:
- assignment-specific
- rawer learning unit
- directly generated from user input

Example:
Question: BST deletion

User rough notes:
"Need successor not predecessor in this implementation"

AI output:
Well-formatted flashcard with explanation.

Metadata includes:
- topic
- related topics
- coursework type
- course

---

## TFC = Topical Flashcard

Definition:
A flashcard summarising learning points aggregated by topic.

Generated by combining topic-relevant content from multiple CFCs.

Example:

Topic:
Binary Search Trees

TFC combines:
- deletion mistakes
- traversal notes
- balancing notes

Characteristics:
- topic-centric
- revision-oriented
- aggregated from many CFCs

# Main Workflow

## Step 1 — Capture Learning

User uploads:
- screenshot/question
- rough notes

Example:
- assignment screenshot
- rough explanation of mistake or insight

---

## Step 2 — AI generates CFC

AI converts rough notes into structured flashcard.

Also extracts metadata.

Example metadata:

```json
{
  "user": "Tauzih",
  "schoolSem": "Year1Sem2",
  "course": "CS2040",
  "type": "CFC",
  "assignmentType": "Tutorial",
  "topics": ["Trees", "BST"],
  "relatedTopics": ["Recursion"]
}
```

CFC stored in database.

---

## Step 3 — Metadata used for organisation

Important architecture decision:

The project uses a TREE only as:
1. organisational abstraction
2. metadata format guide

The tree is NOT the literal database schema.

Example metadata hierarchy:

User
→ SchoolSem
→ Course
→ Type
→ AssignmentType

Example:

Tauzih
→ Year1Sem2
→ IS1108
→ CFC
→ Tutorial

This acts like:
- folder path
- namespace
- indexing scheme

Used for:
- insertion
- retrieval
- JSON formatting

NOT used as literal nested DB.

---

## Step 4 — Generate TFCs

Using metadata:
- sieve all CFCs by topic

Example:

All CFCs tagged:
- Trees
- BST

Aggregated into:

TFC:
BST

This creates topic summaries.

Benefits:
- consolidated revision
- pattern recognition
- recurring mistake detection

---

## Step 5 — Mindmap generation

From TFC metadata:

Each TFC contains:
- current topic
- related topics

These are used to create graph edges.

Example:

BST ↔ Recursion
BST ↔ AVL Trees

Graph:
- nodes = TFCs/topics
- edges = related topics

Not all topics are linked.

Only linked if metadata suggests relation.

---

## Step 6 — AI cross-topic insights

Users click on graph edge.

Example:
BST ↔ Recursion

AI generates insight:

- overlaps
- possible merged exam questions
- conceptual relationships

Insights generated ON DEMAND only.

Reason:
Avoid generating insights for every possible edge.

Optimisation:
Generate only when clicked.

# Database Philosophy

Use relational DB + metadata.

NOT tree DB.

Reason:
Need flexible queries.

Example desired query:

"Get all TFCs under CS2040 regardless of semester, sorted by topic"

SQL example:

```sql
SELECT *
FROM flashcards
WHERE course_code = 'CS2040'
AND flashcard_type = 'TFC'
ORDER BY topic;
```

This is a major reason metadata fields exist.

# Important Architectural Principles

1. Tree is NOT literal DB schema.

Tree only represents:
- organisation
- metadata path

---

2. SQL is primary retrieval mechanism.

Need flexible filtering like:
- all TFCs in CS2040
- all tutorial CFCs
- all notes tagged recursion

---

3. Topic graph is separate from storage.

Do not mix:
- hierarchy
- graph relationships

---

4. AI insights are on-demand only.

Avoid precomputing all edges.

---

5. Metadata is critical.

Everything depends on:
- topic extraction
- related topic extraction
- standardised metadata

Poor metadata = weak system.
